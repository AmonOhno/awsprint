## 1. 目的とコアバリュー

AWS Certified Solutions Architect – Associate（SAA）合格を目的とした個人学習アプリの要件を定義する。

**コアバリュー（差別化要因）**: 既存の問題集アプリとの決定的な違いは、「問題文 → インフラ構成図」への変換を視覚的に反復訓練できる点。正解構成のアーキテクチャ図を提示し、解説文と図の該当箇所をインタラクティブに連動（ハイライト）させることで、文章要件を即座に構成図へ翻訳する「ソリューションアーキテクトの思考回路」を養成する。この図解連動機能を最優先コア機能（Must）とする。

### 用語定義

| 用語 | 定義 |
|------|------|
| SAA | AWS Certified Solutions Architect – Associate 試験 |
| 試験分野 | SAAの4分野（安全なアーキテクチャ／高レジリエンス／高性能／コスト最適化） |
| DiagramSpec | 構成図をノード・エッジ・コンテナで表現する構造化JSON（本アプリ独自形式） |
| 図解連動ハイライト | 解説テキストの語句と構成図のノードを対応づけ、相互に強調表示する機能 |
| MVP | 合格に必要な最小限の機能セット（Phase 1〜3で完成） |

## 3. ペルソナとユーザーストーリー

### 3.1 ペルソナ

エンジニア／これからAWSを学ぶ学習者。座学だけでは「文章で書かれた要件」を構成図に結びつけられず、実務・試験の両方で応用が効かないことに課題を感じている。スキマ時間にPC／タブレットのブラウザで演習し、間違えた問題は図で理解して復習したい。

### 3.2 主要ユーザーストーリー

| ID | ストーリー |
|----|----------|
| US-01 | 学習者として、苦手分野だけを選んでミニテストを解きたい。集中的に弱点を潰すため |
| US-02 | 学習者として、回答後に正解構成のアーキテクチャ図を見たい。文章の要件が実際どう組まれるか理解するため |
| US-03 | 学習者として、なぜ他の選択肢が誤りかを図の該当箇所と紐づけて知りたい。誤答パターンを構造的に覚えるため |
| US-04 | 学習者として、本番同様の65問/130分模試を受けたい。時間配分と合否ラインの感覚を掴むため |
| US-05 | 学習者として、分野別の正解率をグラフで見たい。どこが弱いか一目で把握するため |
| US-06 | 学習者として、間違えた問題をブックマークし後でまとめて復習したい。取りこぼしを防ぐため |
| US-07 | 管理者（本人）として、問題と図データをJSONで一括登録・編集したい。効率よく問題を増やすため |

## 5. アーキテクチャ図表示 詳細要件（コア機能）

FR-06／FR-07 の詳細。実装方式は [feature/diagram_highlight.md](../feature/diagram_highlight.md) で定義する。

### 5.1 表示要件

- 構成図は結果・解説画面の中央に、周辺サービス（VPC・サブネット・AZ・各種サービス）の関係が読み取れる十分な大きさで配置する
- ノードはAWSサービスを表すアイコン＋ラベルを持ち、コンテナ（VPC / AZ / Subnet）は入れ子の枠として描く
- エッジ（接続・通信経路）は向き・ラベル（例: HTTPS, 443）を持てる
- 拡大／縮小・パン操作に対応し、モバイル幅でも破綻しない

### 5.2 解説との連動ハイライト（差別化の核心）

- 解説テキスト中の語句（例:「NATゲートウェイ」）と図中ノードを refId で対応づける
- テキスト→図: 語句にホバー／タップで、対応ノードを強調（発光・拡大・色反転）
- 図→テキスト: ノードにホバー／タップで、関連する解説文をハイライト・スクロール表示
- 誤答の可視化: 「なぜダメか」を、図中の欠落／誤配置箇所（例: NATゲートウェイが無い経路）を赤系で示して説明する

### 5.3 図データのモデル要件

- 図は構造化データ（DiagramSpec: ノード・エッジ・コンテナのJSON）として問題データ内に保持する。静的画像URLも代替として許容する（rendererで切替）
- 各ノードは一意の id を持ち、解説テキストからこのIDで参照される（連動ハイライトの前提）
- スキーマは [data/schema.md](../data/schema.md) で定義する

### 5.4 描画方式の方針

| 方式 | 連動ハイライト適性 / 位置づけ |
|------|------------------------------|
| 構造化JSON＋動的描画（採用: コア） | ◎ ノード単位でイベント・強調が可能。連動ハイライトを実現できる唯一の現実解。本アプリの標準方式 |
| 静的画像（PNG/SVG URL）（代替・移行用） | △ 表示は手堅いが個別ノード連動は困難。図が未整備な問題の暫定表示に使用 |
| LLM自動生成（将来 / Phase 4+） | — 解説文からDiagramSpecを自動生成する将来チャレンジ。MVP対象外 |

### 5.5 図の表示タイミング設定（FR-17）

図を「いつ見せるか」は学習効果に直結するため、ユーザーが設定で切り替えられるようにする。設定はダッシュボード（SCR-01）の学習設定エリアに置き、演習セッション開始時にも上書き選択できる。

| モード | 挙動 / ねらい |
|--------|--------------|
| 解説時のみ（既定） | 回答後の結果・解説画面（SCR-04）でのみ図を表示。まず自力で構成を思い描く力を鍛える、本番想定モード |
| 出題時（ヒント） | 問題演習画面（SCR-03）の問題文下に図を表示。ただし正解・正誤別解説は伏せる。図を手がかりに考える学習初期／苦手分野向けモード |

※ 模擬試験モード（FR-04）では公平性のため、設定に関わらず「解説時のみ」を強制する。

## 7. データ要件

### 7.1 主要エンティティ

| エンティティ | 概要 |
|-------------|------|
| Question | 問題文・分野・選択肢・正解・解説・DiagramSpec・公式リンクを持つ中核 |
| Choice | 選択肢（本文・正誤・その選択肢固有の解説） |
| DiagramSpec | ノード・エッジ・コンテナで構成図を表す構造化データ |
| Attempt | 1問の解答記録（選択・正誤・所要時間・日時）。正解率算出の元 |
| ExamSession | 模試1回分（出題セット・残り時間・スコア・合否） |
| Bookmark | 要復習登録（問題への参照） |
| UserSettings | 図の表示タイミング（FR-17）、合格目安ライン |

連動ハイライトの鍵は、**解説文の refIds と DiagramSpec の node.id が一致すること**。
完全なスキーマは [data/schema.md](../data/schema.md)、実データは `data/questions.sample.json` を参照。

## 8. 非機能要件

| ID | 分類 | 要件 |
|----|------|------|
| NFR-01 | 性能 | 問題遷移は体感1秒以内、図描画は2秒以内。100問規模でも快適に動作 |
| NFR-02 | 対応環境 | PC/タブレットの最新Chrome・Safari・Edge。レスポンシブ対応（当面Webのみ） |
| NFR-03 | 可用性/保守性 | 問題データはコード外（リポジトリ内JSON）で管理し、非エンジニアでも追記可能な構造にする |
| NFR-04 | セキュリティ | **（v2.0改訂）** サーバー・DB・秘密情報を持たないため認証は不要。学習履歴は端末内に閉じ、外部送信しない。将来バックエンドを追加する場合は認証を必須とする |
| NFR-05 | データ整合性 | インポート時にzodでスキーマ検証。refIdとnode.idの不整合を警告する |
| NFR-06 | コスト | **（v2.0改訂）ランニングコスト0円を絶対条件とする。** 無料枠依存の従量課金サービスは採用しない（枠超過・期限切れによる課金リスクを排除） |
| NFR-07 | アクセシビリティ | 図の情報はテキスト解説でも同等に得られる（色のみに依存しない） |

## 9. 制約・前提条件

- 開発・運用は個人。低コスト・低運用負荷を最優先する
- **（v2.0改訂）** インフラはランニングコスト0円の静的ホスティング（GitHub Pages）とし、バックエンドは持たない。v1.0 の「AWS学習を兼ねてAWSサーバーレスを採用する」方針は撤回（理由は [architecture/overview.md](../architecture/overview.md) 参照）。AWSはアプリの**教材**として扱う
- 問題文・解説・図データの著作権はオリジナル作成とし、AWS公式ドキュメントは引用リンクに留める。本試験の設問そのものの転載は行わない（リポジトリをPublicにする前提条件でもある）
