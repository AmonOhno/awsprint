1. 設計方針

  • サーバーレス優先: 個人運用のアイドルコスト最小化（NFR-06）と、AWS学習の実地化を兼ねる。
  • データ駆動: 問題・図はコード外のJSON/DBで管理し、追記だけで拡張できる（NFR-03）。
  • 図はコア: 連動ハイライト（FR-07）を実現するため、図を構造化データ＋動的描画で扱うことを設計の中心に置く。
  • 段階的移行: Phase 2まではローカルJSON＋ブラウザ保存で素早く動かし、Phase 3以降でAWSバックエンドへ拡張する（差し替え可能な抽象化）。

4. アーキテクチャ図レンダリング設計 ★コア

要件定義書 第5章（FR-06/FR-07）の実装方式を定義する。本アプリの差別化価値を担う中心モジュール。

4.1 方式：DiagramSpec → React Flow

DB/JSONの DiagramSpec（宣言的なノード・エッジ・コンテナ）を、変換関数でReact Flowの nodes/edges にマッピングして描画する。各ノードは id を保持し、解説側の refId と一致させる。

4.2 DiagramSpec スキーマ

type DiagramSpec = {
  renderer: "spec" | "image";      // spec=動的描画 / image=静的URL(代替)
  imageUrl?: string;               // renderer=image のとき
  containers?: Container[];        // VPC / AZ / Subnet などの入れ子枠
  nodes: DiagramNode[];
  edges: DiagramEdge[];
};

type Container = {
  id: string;
  type: "VPC" | "AZ" | "PublicSubnet" | "PrivateSubnet" | "Region";
  label: string;
  parent?: string;                 // 入れ子（AZ ⊂ VPC など）
};

type DiagramNode = {
  id: string;                      // ★ 解説の refId と一致させる
  service: AwsService;             // "EC2" | "S3" | "NATGateway" ...
  label: string;
  parent?: string;                 // 所属コンテナ
  state?: "normal" | "faulty";     // 誤答説明で欠陥箇所を赤表示
};

type DiagramEdge = {
  from: string; to: string;
  label?: string;                  // "HTTPS 443" など
  directed?: boolean;
};

4.3 解説連動ハイライトの仕組み（FR-07）

  1. 解説文はミニマークアップ <ref id="nat">NATゲートウェイ</ref> を含む。パーサがこれを <RefSpan nodeId> に変換。
  2. 共有ストア useHighlight() が activeId を保持。
  3. テキスト→図: RefSpan の hover/tap で setActive(id) → 対応ノードが発光。
  4. 図→テキスト: ノードの hover/tap で同ストア更新 → 対応RefSpanを強調・スクロール。
// features/diagram/useHighlightLink.ts
const useHighlight = create<{ activeId?: string; setActive(id?: string): void }>(
  (set) => ({ activeId: undefined, setActive: (id) => set({ activeId: id }) })
);
// ノード側: className に activeId===node.id で "ring/glow" を付与
// RefSpan側: 同条件で背景ハイライト＋scrollIntoView相当のフォーカス

4.4 補完方式とフォールバック

  • Mermaid: 連動が不要な補足図はMermaidテキストでも描画可（renderer拡張）。
  • 静的画像: 図が未整備の問題は renderer:"image" で暫定表示し、後日specへ移行。
  • AWSアイコン: 公式「AWS Architecture Icons」相当のサービス別アイコンをカスタムノードで表示（サービス名→アイコンのマップを一元管理）。

4.5 図の表示タイミング（FR-17）

同一の DiagramView を、設定 settings.diagramTiming に応じて出題画面（SCR-03）と結果画面（SCR-04）のどちらにマウントするかを切り替える。図コンポーネントは表示位置に依存しないため、追加実装は「どこに置くか」の分岐のみ。

  • "explanation"（既定）: 結果画面でのみ描画。連動ハイライトも解説と同時に有効化。
  • "question": 出題画面に hintMode で描画。choice.correct / explanation / node.state="faulty" は回答確定まで非表示にマスクする。
  • 模試（FR-04）は examMode のとき設定を無視し常に "explanation" 相当に固定。

6. データベース設計（DynamoDB）

6.1 シングルテーブル設計

1テーブル SaaApp にPK/SKで全エンティティを格納。GSI1で分野別・ユーザー別アクセスを賄う。

エンティティ PK SK / GSI1
QuestionQUESTION#{id}META / GSI1PK=DOMAIN#{domain}
AttemptUSER#{uid}ATTEMPT#{ts}#{qid}
ExamSessionUSER#{uid}EXAM#{ts}
BookmarkUSER#{uid}BOOKMARK#{qid}

6.2 主要アクセスパターン

  • 分野別の問題取得 → GSI1（GSI1PK=DOMAIN#SECURE_ARCH）をQuery。
  • ユーザーの解答履歴 → PK=USER#uid, SK begins_with ATTEMPT#。
  • 分野別正解率 → 履歴を集計（件数が増えたら日次で集計項目に事前集約）。

7. データモデル（TypeScript型定義）

正規のソース。zodスキーマも同定義から生成し、インポート時の検証（NFR-05）に用いる。

type Domain =
  | "SECURE_ARCH"      // 安全なアーキテクチャの設計
  | "RESILIENT"        // 高レジリエンスアーキテクチャの設計
  | "HIGH_PERF"        // 高性能アーキテクチャの設計
  | "COST_OPT";        // コスト最適化アーキテクチャの設計

type Choice = {
  id: string;
  text: string;
  correct: boolean;
  why?: string;                    // 誤答/正答の理由（FR-08）
};

type Explanation = {
  summary: string;                 // <ref id="..."> を含むミニマークアップ
  refIds: string[];                // 図ノードと対応（FR-07）
};

type Question = {
  id: string;
  domain: Domain;
  type: "single" | "multiple";
  stem: string;
  choices: Choice[];
  explanation: Explanation;
  diagram?: DiagramSpec;           // 図（コア）
  docLinks?: { title: string; url: string }[];
  tags?: string[];
};

type Attempt = {
  questionId: string;
  domain: Domain;
  selected: string[];
  correct: boolean;
  elapsedSec: number;
  answeredAt: string;              // ISO8601
};

type ExamSession = {
  id: string;
  questionIds: string[];           // 65問
  startedAt: string;
  durationSec: number;             // 130*60
  attempts: Attempt[];
  scorePct?: number;
  passed?: boolean;                // 合格ライン判定
};

type UserSettings = {
  diagramTiming: "explanation" | "question";  // FR-17: 図の表示タイミング（既定 explanation）
  passLinePct: number;                         // 合格目安ライン（既定 72）
};

type ProgressSummary = {
  totalAnswered: number;
  overallPct: number;
  byDomain: Record<Domain, { answered: number; correctPct: number }>;
};

8. 採点・進捗ロジック

  • 採点（FR-03）: 単一選択は一致で正解。複数選択は選択集合が正解集合と完全一致で正解（部分点なし＝本番準拠）。
  • 分野別正解率（FR-11）: correctPct = 正解Attempt数 / 該当分野Attempt数。レーダーチャートに反映。
  • 模試スコア（FR-04）: 得点率を算出し、合格目安ライン（設定値・既定72%相当）で合否判定。未回答は誤答。
  • 弱点抽出（SCR-06）: 分野別正解率の最低2分野を「重点復習」として提示。
