1. はじめに

1.1 目的

本書は、AWS Certified Solutions Architect – Associate（以下 SAA）合格を目的とした個人学習アプリの詳細な機能・非機能要件を定義する。元の要件定義書（背景メモ）を、開発に着手できる粒度まで具体化し、各要件に一意なIDと受け入れ基準を付与する。実装方式は姉妹文書「技術仕様書（DOC-SPEC-001）」で定義する。

1.2 本アプリのコアバリュー（差別化要因）

既存の問題集アプリとの決定的な違いは、「問題文 → インフラ構成図」への変換を視覚的に反復訓練できる点にある。単なる正誤判定ではなく、正解構成のアーキテクチャ図を提示し、解説文と図の該当箇所をインタラクティブに連動（ハイライト）させることで、文章要件を即座に構成図へ翻訳する「ソリューションアーキテクトの思考回路」を養成する。この図解連動機能を本アプリの最優先コア機能（Must）と位置づける。

3. ペルソナとユーザーストーリー

3.1 ペルソナ（初期フェーズ）

エンジニア／これからAWSを学ぶ学習者。座学だけでは「文章で書かれた要件」を構成図に結びつけられず、実務・試験の両方で応用が効かないことに課題を感じている。スキマ時間にPC／タブレットのブラウザで演習し、間違えた問題は図で理解して復習したい。

3.2 主要ユーザーストーリー

ID ストーリー
US-01学習者として、苦手分野だけを選んでミニテストを解きたい。集中的に弱点を潰すため。
US-02学習者として、回答後に正解構成のアーキテクチャ図を見たい。文章の要件が実際どう組まれるか理解するため。
US-03学習者として、なぜ他の選択肢が誤りかを図の該当箇所と紐づけて知りたい。誤答パターンを構造的に覚えるため。
US-04学習者として、本番同様の65問/130分模試を受けたい。時間配分と合否ラインの感覚を掴むため。
US-05学習者として、分野別の正解率をグラフで見たい。どこが弱いか一目で把握するため。
US-06学習者として、間違えた問題をブックマークし後でまとめて復習したい。取りこぼしを防ぐため。
US-07管理者（本人）として、問題と図データをJSONで一括登録・編集したい。効率よく問題を増やすため。

5. アーキテクチャ図表示 詳細要件（コア機能）

FR-06／FR-07 を掘り下げる。本章は本アプリの心臓部であり、実装方式は技術仕様書 第4章で定義する。

5.1 表示要件

  • 構成図は結果・解説画面の中央に、周辺サービス（VPC・サブネット・AZ・各種サービス）の関係が読み取れる十分な大きさで配置する。
  • ノードはAWSサービスを表すアイコン＋ラベルを持ち、コンテナ（VPC / Availability Zone / Subnet）は入れ子の枠として描く。
  • エッジ（接続・通信経路）は向き・ラベル（例: HTTPS, 443）を持てる。
  • 拡大／縮小・パン操作に対応し、モバイル幅でも破綻しない。

5.2 解説との連動ハイライト（差別化の核心）

  • 解説テキスト中の語句（例:「NATゲートウェイ」）と図中ノードを refId で対応づける。
  • テキスト→図: 語句にホバー／タップで、対応ノードを強調（発光・拡大・色反転）。
  • 図→テキスト: ノードにホバー／タップで、関連する解説文をハイライト・スクロール表示。
  • 誤答の可視化: 「なぜダメか」を、図中の欠落／誤配置箇所（例: NATゲートウェイが無い経路）を赤系で示して説明する。

5.3 図データのモデル要件

  • 図はDBに構造化データ（DiagramSpec: ノード・エッジ・コンテナのJSON）として保持する。静的画像URLも代替として許容する（rendererで切替）。
  • 各ノードは一意の id を持ち、解説テキストからこのIDで参照される（連動ハイライトの前提）。
  • 具体的なスキーマは技術仕様書 4.2 節で定義する。

5.4 描画方式の方針

方式 連動ハイライト適性 / 位置づけ
構造化JSON＋動的描画
（採用: コア）		◎ ノード単位でイベント・強調が可能。連動ハイライトを実現できる唯一の現実解。本アプリの標準方式。
静的画像（PNG/SVG URL）
（代替・移行用）		△ 表示は手堅いが個別ノード連動は困難。図が未整備な問題の暫定表示に使用。
LLM自動生成
（将来 / Phase 4+）		— 解説文からDiagramSpecを自動生成する将来チャレンジ。MVP対象外。

5.5 図の表示タイミング設定（FR-17）

図を「いつ見せるか」は学習効果に直結するため、ユーザーが設定で切り替えられるようにする。設定はダッシュボード（SCR-01）の学習設定エリアに置き、演習セッション開始時にも上書き選択できる。

モード 挙動 / ねらい
解説時のみ
（既定）		回答後の結果・解説画面（SCR-04）でのみ図を表示。まず自力で構成を思い描く力を鍛える、本番想定モード。
出題時（ヒント）問題演習画面（SCR-03）の問題文下に図を表示。ただし正解・正誤別解説は伏せる。図を手がかりに考える学習初期／苦手分野向けモード。

※ 模擬試験モード（FR-04）では公平性のため、設定に関わらず「解説時のみ」を強制する。

8. 非機能要件

ID 分類 要件
NFR-01性能問題遷移は体感1秒以内、図描画は2秒以内。100問規模でも快適に動作。
NFR-02対応環境PC/タブレットの最新Chrome・Safari・Edge。レスポンシブ対応（当面Webのみ）。
NFR-03可用性/保守性問題データはコード外（JSON/DB）で管理し、非エンジニアでも追記可能な構造にする。
NFR-04セキュリティ管理機能は認証で保護。初期は単一ユーザー前提だが、将来のマルチユーザー化を阻害しない設計とする。
NFR-05データ整合性インポート時にスキーマ検証。refIdとnode.idの不整合を警告する。
NFR-06コスト個人利用のため無料枠中心。サーバーレス従量課金でアイドル時コストを最小化。
NFR-07アクセシビリティ図の情報はテキスト解説でも同等に得られる（色のみに依存しない）。

9. 制約・前提条件

  • 開発・運用は個人。低コスト・低運用負荷を優先する。
  • AWSの学習を兼ねるため、インフラは可能な範囲でAWSサーバーレスを採用する（技術仕様書で定義）。
  • 問題文・解説・図データの著作権はオリジナル作成とし、AWS公式ドキュメントは引用リンクに留める。
  • 本試験の設問そのものの転載は行わない。
